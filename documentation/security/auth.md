# RustDB Server Authentication and Authorization

This document describes the authentication and authorization system used by `rustdb-server`. It covers how the middleware works, how to store and validate API keys, how role-based access control is enforced, and how the system is tested.

## Overview

The authentication system lives under `rustdb-server/src/security/auth/` and consists of the following components:

- `api_key.rs` — API key model and role definitions.
- `hashing.rs` — secure password hashing and verification using Argon2.
- `verification.rs` — bearer token parsing and key verification logic.
- `middleware.rs` — Axum middleware to authenticate requests, enforce rate limiting, and write audit events.
- `rate_limit.rs` — simple in-memory token bucket rate limiting.
- `audit.rs` — immutable audit logging.
- `rbac.rs` — permission checks for role-based access control.
- `macros.rs` — convenience macro `require_role!` for handlers.

## Data Model

The system assumes API key records are stored in a server-side store and retrieved by key ID.

```rust
pub struct ApiKey {
    pub id: String,
    pub key_hash: String,
    pub role: ApiRole,
    pub created_at: i64,
    pub revoked: bool,
}

pub enum ApiRole {
    Admin,
    ReadWrite,
    ReadOnly,
}
```

The API key store interface is defined in `middleware.rs`:

```rust
pub trait ApiKeyStore {
    fn find_by_id(&self, id: &str) -> Result<Option<ApiKey>, ()>;
}
```

### Roles

- `Admin` — full access
- `ReadWrite` — read and write access
- `ReadOnly` — read-only access

## API Key Storage and Generation

Do not store raw API key secrets.
Use `hash_api_key(secret)` to hash the raw secret before persisting.

```rust
let secret = "super-secret";
let hash = hash_api_key(secret)?;
```

To validate a secret against a stored hash, use:

```rust
let ok = verify_api_key(&stored_hash, raw_secret);
```

### Hashing implementation

The project uses Argon2 with secure defaults:
- `Argon2::default()`
- random salt generated by `rand_core::OsRng`
- constant-time verification

A malformed hash string is treated as invalid without panicking.

## Request Format

Authenticated requests must include an `Authorization` header with the following format:

```
Authorization: Bearer <key_id>.<secret>
```

Example:

```
Authorization: Bearer abc123.DEF456
```

The middleware parses the header with `split_bearer()` and splits the token into `key_id` and `secret`.

## Middleware Behavior

The main middleware function is `auth_middleware()` in `middleware.rs`.
It performs the following steps in order:

1. If development mode is enabled via `set_dev_mode(true)`, allow the request without authentication.
2. Extract the `Authorization` header and parse it with `split_bearer()`.
3. Apply rate limiting using `check_rate_limit(key_id)`.
4. Use the configured `ApiKeyStore` to look up the key by ID.
5. Verify the extracted secret using `verify_key(secret, &stored_key)`.
6. If verification succeeds, insert an `AuthContext` into the request extensions.
7. Write an audit event for success or failure.

### Auth Context

On success, `AuthContext` is inserted into request extensions:

```rust
pub struct AuthContext {
    pub key_id: String,
    pub role: ApiRole,
    pub issued_at: i64,
}
```

Handlers can retrieve this context from `req.extensions()`.

## Rate Limiting

Rate limiting is implemented in `rate_limit.rs` as an in-memory token bucket keyed by API key ID.

- `MAX_TOKENS = 100`
- `REFILL_PER_SEC = 10`

The function `check_rate_limit(key_id)` returns `true` if the request is allowed and decrements the bucket. If the bucket is empty, it returns `false`.

This is a simple local limiter and should be replaced with Redis or another distributed store in a cluster.

## Audit Logging

Audit events are written by `write_audit()` in `audit.rs`.

A log line is appended to `audit.log` in the local working directory:

```
<timestamp>|<key_id>|<action>|<ip>|<success>\n
```

The audit logger intentionally avoids panicking if the file cannot be opened.

## Role-Based Authorization

There are two permission utilities:

- `verification::has_permission(user_role, required_role)` — strict role hierarchy.
- `rbac::has_permission(role, Permission)` — permission-level checks.

The `Permission` enum includes:

```rust
pub enum Permission {
    Read,
    Write,
    Admin,
}
```

### Macro helper

`require_role!(req, Permission::Write)` is provided by `macros.rs`.
It checks the `AuthContext` in `req.extensions()` and returns `UNAUTHORIZED` or `FORBIDDEN` if the request is not allowed.

Example handler:

```rust
fn handler(req: Request<Body>) -> Result<String, StatusCode> {
    require_role!(req, Permission::Write);
    Ok("allowed".to_string())
}
```

## Integrating the Auth Middleware

Example Axum integration:

```rust
use std::sync::Arc;
use axum::{Router, routing::get, extract::State};
use rustdb_server::security::auth::middleware::{auth_middleware, ApiKeyStore};

struct MyStore;
impl ApiKeyStore for MyStore {
    fn find_by_id(&self, id: &str) -> Result<Option<ApiKey>, ()> {
        // lookup key in database
    }
}

let store = Arc::new(MyStore);
let app = Router::new()
    .route("/data", get(my_handler))
    .layer(axum::middleware::from_fn_with_state(store.clone(), move |State(store): State<Arc<MyStore>>, req, next| {
        let store = store.clone();
        async move { auth_middleware(req, next, store).await }
    }));
```

## Validation and Testing

The auth system is covered by tests in `rustdb-server/tests/security/auth.rs`.
The `security` test module also loads crypto tests under `tests/security/crypto/*`.

Important validation cases include:

- valid key allows request
- missing `Authorization` header is rejected
- invalid bearer format is rejected
- unknown key ID is rejected
- wrong secret is rejected
- revoked key is rejected
- dev mode allows bypass
- rate-limit exhaustion rejects requests
- `split_bearer()` parsing
- hashing/verifying API keys
- permission checks for RBAC and role hierarchy
- `require_role!` macro acceptance and denial

Run the suite with:

```bash
cargo test --test security -- --nocapture
```

## Security Considerations

### What is protected

- The system protects every request with an API key unless dev mode is enabled.
- The stored API key secret is never compared in plaintext; only the Argon2 hash is used.
- Revoked keys are rejected before any verification attempt.
- Request rate limiting is enforced before authentication success.
- Audit events are written for every authentication attempt.

### Threat model notes

- If an attacker obtains a key ID alone, the system still rejects requests without the secret.
- If an attacker obtains a key hash, the hash is not directly usable unless Argon2 inversion succeeds.
- The current rate limiter is local and not distributed; in a multi-instance deployment, it must be replaced by a shared rate-limit store.
- The middleware does not currently verify TLS client IP or additional headers beyond `x-forwarded-for` for audit.

### Suggested hardening

- Use HTTPS for all requests.
- Store API keys and secrets in a secure database.
- Rotate keys periodically using `rotation.rs` and soft revoke old keys.
- Replace the in-memory rate limiter with Redis or another centralized service for distributed deployments.
- Ensure `audit.log` is protected and shipped to a secure log sink.
- Do not enable `set_dev_mode(true)` in production.

## Example workflow

1. Create a new API key secret.
2. Hash it with `hash_api_key(secret)`.
3. Persist an `ApiKey` record with `id`, `key_hash`, `role`, and `created_at`.
4. Configure an `ApiKeyStore` implementation and wire it into middleware.
5. Send requests using `Authorization: Bearer <id>.<secret>`.
6. Use `require_role!(req, Permission::Write)` or `rbac::has_permission(...)` in handlers.
7. Run `cargo test --test security` to validate behavior.
