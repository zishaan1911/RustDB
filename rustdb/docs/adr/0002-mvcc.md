# ADR 0002: MVCC Isolation

## Status

Accepted

## Context

We needed to choose an isolation level and implementation strategy. Options:
- **Serializable**: Highest safety but worst performance
- **MVCC with Snapshot Isolation**: Good performance with acceptable safety
- **Lock-based REPEATABLE READ**: Traditional but blocks readers

## Decision

Implement MVCC with Snapshot Isolation as the default.

## Rationale

1. **Read Performance**: Readers never block on writers
2. **Write Performance**: Optimistic locking reduces contention
3. **Proven Approach**: PostgreSQL, MySQL use similar techniques
4. **Acceptable Semantics**: Snapshot isolation sufficient for most workloads
5. **Implementability**: Standard techniques available

## Consequences

### Positive
- Excellent concurrency for read-heavy workloads
- Non-blocking reads
- Memory efficient with vacuum

### Negative
- Phantom reads possible (acceptable trade-off)
- Write conflicts require retry
- More complex implementation

## Alternatives Considered

1. **Serializability**: Too strict, would hurt performance
2. **Simple Locking**: Good for certain workloads but blocking
